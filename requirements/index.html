Requirements Overview

Requirements connect the problem, system architecture, and verification plan.

The final PICKS framework organizes requirements around inventory visibility, traceability, kitting, compliance, interface readiness, enterprise integration, and user adoption.

Stakeholder Needs

What PICKS Must Support

SNR-1

Inventory Visibility

Provide clearer inventory status, lifecycle workflow support, and project-level visibility.

SNR-2

Traceability

Maintain chain-of-custody and history for critical parts, components, kits, and related records.

SNR-3

Kitting

Support controlled kit creation, BOM validation, configuration tracking, and material issuance.

SNR-4

Compliance Artifacts

Connect inventory and release workflows to contract-driven documentation and audit evidence.

SNR-5

Interface Readiness

Define how external data is retrieved, mapped, displayed, protected, and handled during failures.

SNR-6

Enterprise Integration

Exchange data with enterprise systems without duplicating future ERP responsibilities.

SNR-7

Adoption and Training

Support user guidance and onboarding so the architecture can be adopted across lab contexts.

SYS-1.2.1

Show program-level inventory status

Share information across systems

PICKS needs to move useful information between ERP, Workday, inventory, and lab systems.

Workflow execution should continue when an enterprise system is temporarily unavailable.

SYS-6.5

Guide the user

Prioritization

How the team thought about priority

Requirements were evaluated using mission alignment, MVP operational criticality, and complexity penalty. This keeps the final architecture focused on high-value capabilities while recognizing integration and implementation difficulty.

Mission Alignment Does it support visibility, traceability, compliance, and standardized workflows?
MVP Criticality Is it needed for core receiving, inventory, kitting, or lifecycle work?
Complexity Penalty Does it introduce integration, development, or adoption difficulty?
